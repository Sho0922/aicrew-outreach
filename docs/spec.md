# 東京都内 中小企業 営業アタックリスト（aicrew AI・DX 自動化）— 仕様

作成日: 2026-06-10
対象事業: **aicrew**「PoC止まりのAI・DXを現場で動く成果へ」生成AI / DX 導入伴走支援
目的: 東京都内のレガシー中小企業へ、業務自動化（生成AI + RPA + OCR）導入伴走の営業メールをアタックするためのターゲットリストを、登記オープンデータからクローリングして生成する。

前回の Hempets（動物病院チャネル）アタックリストと同じ思想——**クロール → 正規化 → AI 的スコアリング → CSV/xlsx 出力**——を踏襲する。Hempets は Google Maps 由来の病院データを `hospitals` テーブルに取り込む構成だったが、本件は B2B のため**法人登記オープンデータ（gBizINFO）**を一次ソースにし、スタンドアロンの CSV/xlsx として出力する。

---

## 1. ターゲット定義

| 条件 | 値 | データ上の扱い |
| --- | --- | --- |
| 所在地 | 東京都内 | gBizINFO `prefecture=13` で確実に絞れる |
| 従業員数 | 100人以下 | gBizINFO `employee_number_from/to`（既定 20〜100） |
| 年商 | 10〜30億円 | **登記データに無い**。財務があれば実測、無ければ従業員数×業種ベンチで推定 |
| 業種 | 製造 / 卸売・商社 / 建設・設備 / 物流・運輸 / 不動産管理 | 取得後にキーワード分類（`classifyIndustry`） |
| 性質 | レガシー（FAX・紙・SaaS乱立で自動化余地大） | **登記データに無い**。業種・設立年・自社サイト有無から `legacy_score` を推定 |

> 既定の資本金レンジ `1千万〜3億円` は「年商10〜30億円」の代理指標。年商そのものは確度を `revenue_confidence`（reported / estimated）で明示し、**メール送信前に人手検証**する前提。

---

## 2. データソース

- **gBizINFO REST API v1** — `https://info.gbiz.go.jp/hojin/v1/hojin`
  - 認証: HTTP ヘッダ `X-hojinInfo-api-token: <トークン>`
  - トークン: 無料・無期限。`https://info.gbiz.go.jp/` ログイン → マイページで発行。
  - 取得フィールド: 法人番号 / 法人名 / 所在地 / 郵便番号 / 代表者 / 設立年月日 / 資本金 / 従業員数 / 事業概要 / 自社URL ほか。財務（売上高）は提出のある法人のみ。
- 補足: 年商・現場の業務実態（FAX/SaaS 利用）は登記データに無いため、後段で各社サイト・求人等を人手 or 追加クロールで補強する。

---

## 3. パイプライン

```
gBizINFO 検索（東京都 × 従業員20-100 × 資本金1千万-3億 × 現存）
  → ページング取得（最大ページ・delay 付き、レート配慮）
  → classifyIndustry: 法人名/事業概要/業種から 5 バケットへ分類（非該当は除外）
  → toProspect:
       ・年商 = 財務あれば実測 / 無ければ 従業員数 × 業種別1人あたり年商で推定
       ・legacy_score(0-100) = 社歴 + 自社サイト有無 + 業種ベース + 規模
       ・priority_tier(S/A/B/C) = 年商帯一致 × 100人以下 × レガシー度
  → dedupeAndRank: 法人番号で重複排除 → ティア→レガシー度で降順
  → CSV(UTF-8 BOM) + xlsx 出力（data/prospects.csv / .xlsx）
```

実装:
- `src/lib/gbizinfo.ts` — API クライアント・分類・スコアリング・出力。
- `scripts/crawl-prospects.ts` — CLI（`pnpm crawl-prospects`）。

## 4. スコアリング基準

### legacy_score（0-100, 高いほど自動化余地大）
- 設立が古い（社歴長い）: 最大 +35（50年で満点）
- 自社サイト未検出: +25（デジタル化遅れの兆候）
- 対象レガシー業種: +15〜25（製造/卸=25, 建設=22, 物流=20, 不動産管理=15）
- 規模 20〜100人で属人化: +15

### priority_tier
| ティア | 条件 |
| --- | --- |
| **S** | 年商10-30億帯 × 100人以下 × legacy_score ≥ 70 |
| **A** | 年商帯一致 × 100人以下 × legacy_score ≥ 55 |
| **B** | 100人以下 かつ（年商帯一致 or legacy_score ≥ 60） |
| **C** | 上記以外（100人超/不明 など） |

### automation_opportunity / recommended_hook
業種ごとに自動化の切り口を定義し、aicrew 資料の数値（処理時間 -88% / 3年最大 3,675万円コスト削減 / AI導入コスト 1/10 / 伴走型・定着率100%）に紐づくフックを生成。

| 業種 | 自動化の切り口 |
| --- | --- |
| 製造 | FAX/電話受発注・生産管理Excel・在庫照合の自動化、図面/仕様問合せの生成AI対応 |
| 卸売・商社 | FAX受注・見積/請求の手入力を OCR+生成AI で自動起票、一次対応自動化 |
| 建設・設備 | 見積/工程/原価の帳票自動生成、現場写真・日報の生成AI整理 |
| 物流・運輸 | 配車/伝票/請求処理の自動化、追跡・問合せの AI チャット化 |
| 不動産管理 | 契約更新・入居者問合せ対応の自動化、紙台帳のデータ化 |

優先度 / 法人名 / 業種 / 都道府県 / 市区町村 / 住所 / 郵便番号 / 代表者 / URL / 自社サイト有無 / 設立年 / 社歴 / 資本金 / 従業員数 / 推定年商(億円) / 年商確度 / 10-30億帯 / レガシー度 / 自動化余地 / 提案フック / 優先度の根拠 / 想定接触手段 / 事業概要 / 法人番号 / ソース / 取得日時

ヘッダのみのテンプレートは `data/prospects.template.csv`。

## 6. 実行手順

```bash
# 1. トークンを設定
echo 'GBIZINFO_API_TOKEN=xxxxx' >> .env.local

# 2. 設定確認（API は叩かない）
pnpm crawl-prospects:dry

# 3. 生成（既定: 東京都 × 20-100人 × 資本金1千万-3億）
pnpm crawl-prospects

# 例: S/A ティアだけ、ページ多め
pnpm crawl-prospects --tier=S,A --max-pages=25 --out=data/tokyo_smb
```

主なオプション: `--pref` `--emp-from/--emp-to` `--cap-from/--cap-to` `--max-pages` `--delay` `--tier` `--out`。

## 7. 限界と運用上の注意（誠実性）

- **年商と「FAX/SaaS 利用」は推定**。登記オープンデータには無い。Tier S/A でも、年商確度が `estimated` の行は送信前に各社サイト/決算/帝国データバンク等で要検証。
- gBizINFO の従業員数・財務は**未更新/未提出**の法人がある。`null` は除外せずティア C 寄りに落とす。
- レート制限に配慮し `--delay` を確保。大量取得時は資本金/従業員のサブレンジに分割して複数回実行する。
- 生成された実リスト（`data/*.csv` / `*.xlsx`）は `.gitignore` 済み。**個人情報・営業リストはコミットしない**。
- B2B 営業メールは特定電子メール法（オプトアウト導線の明示等）を遵守。配信停止導線は既存 Hempets メール基盤（`mail_optouts` / 自動フッタ）の運用に合わせる。
